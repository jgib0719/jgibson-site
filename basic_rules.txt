# Basic Rules for CCNA Study Guide Website Development

## File Naming and Organization
1. All section data files follow pattern: `section[X]-data.js` where X is section number
2. All section pages follow pattern: `section[X].html` where X is section number
3. Data keys in JavaScript files should NOT include "Grid" suffix - this is added automatically by section-loader.js
4. Each section has corresponding detail files: `section[X]-details.js`
5. CSS classes and IDs should follow kebab-case convention

## Data Structure Consistency
1. Each section data file must have a metadata object with:
   - sectionNumber: number
   - title: string
   - subsections: object with keys matching HTML div IDs (without "Grid" suffix)
   - totalTopics: computed property summing subsection counts

2. HTML div elements that load content must have IDs ending in "Grid"
3. The div ID (minus "Grid") must exactly match a key in the corresponding data file

## JavaScript Architecture
1. section-loader.js is the shared loader for all sections
2. It expects data keys WITHOUT "Grid" suffix and adds "Grid" when looking for DOM elements
3. All sections register with CCNA_SECTION_REGISTRY for progress tracking
4. Event-driven architecture: sections dispatch 'sectionDataLoaded' events

## CSS Organization
1. site.css contains all styles - no external framework dependencies
2. Avoid duplicate CSS rules - always check for existing styles before adding new ones
3. Modal styles are centralized and shared across all pages
4. Use CSS custom properties (variables) for consistent theming
5. Mobile-first responsive design approach

## Content Standards
1. Each topic card must have: title, icon (FontAwesome), description, and visual (SVG)
2. SVG visuals should be self-contained and accessible
3. All interactive elements need proper ARIA attributes
4. Color contrast must meet WCAG AA standards

## Testing and Quality
1. Always test section pages after data structure changes
2. Verify progress tracking works across all sections
3. Check responsive design on multiple screen sizes
4. Validate HTML and CSS regularly
5. Test keyboard navigation and screen reader compatibility

## Git Workflow
1. Make incremental commits with clear messages
2. Test changes locally before committing
3. Document any breaking changes in commit messages
4. Keep documentation files updated with codebase changes
